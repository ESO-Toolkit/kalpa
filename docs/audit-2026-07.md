# Repository Audit — July 2026

A full-repo review of Kalpa covering the Rust backend (`src-tauri/`), the React
frontend (`src/`), the Pack Hub Cloudflare Worker (`backend/eso-packs-worker/`),
CI/CD, build configuration, e2e tests, and docs. Findings are ordered by
severity; every item carries a `file:line` reference against the tree at the
time of the audit. No code was changed as part of this audit — this document is
the deliverable.

**Bottom line:** the codebase is unusually well-hardened for a beta-stage
desktop app — the classic Tauri failure modes (zip-slip, path traversal, loose
CSP, broad capabilities) are all handled deliberately and well. The two
findings that deserve prompt attention are both outside the app itself: the
Pack Hub worker does not enforce anonymity server-side, and releases are
published without any test gate.

---

## Strengths worth preserving

These came up repeatedly during review and are the reason most findings below
are minor:

- **Zip extraction** (`src-tauri/src/installer.rs`) is a model implementation:
  `enclosed_name()` for traversal safety, symlink entries skipped via the unix
  mode bit, a declared-size zip-bomb cap, and rollback of only newly-created
  top-level folders on failure.
- **Path confinement**: every caller-supplied addons path is validated against
  canonicalized managed state (`commands.rs` `validate_addons_path` /
  `require_allowed_path`); relative paths reject `..`, absolute paths, and
  Windows-forbidden characters including `:` (blocks NTFS ADS tricks); the
  uploader blocks UNC/device-namespace paths and confines log reads to the ESO
  `Logs` directory with the TOCTOU window deliberately closed.
- **Webview hardening**: strict production CSP (`script-src 'self'`, no
  `unsafe-eval`), least-privilege capabilities with `opener` scoped to an
  explicit URL allowlist, all outbound HTTP routed through the Rust side.
- **Supply chain**: every GitHub Action is SHA-pinned with version comments;
  workflow `permissions:` are minimally scoped; cargo-audit and clippy
  `-D warnings` run in CI.
- **Frontend discipline**: raw `invoke()` appears in only two files — everything
  else goes through the `invokeResult`/`invokeOrThrow` wrappers with
  user-facing error mapping (`src/lib/tauri.ts`); zero `@ts-ignore`, near-zero
  `any`, no `dangerouslySetInnerHTML` anywhere (remote ESOUI descriptions are
  parsed into React elements, XSS-safe by construction); settings persistence
  is crash-atomic with rollback (`src/lib/store.ts`).
- **Rust test depth**: 646 test functions across 44 files, concentrated in
  exactly the risk-bearing parsers (encounter-log events/encoding,
  SavedVariables scrubbing, file hashing).
- **Worker basics**: D1 access is fully parameterized (no injection), the admin
  key uses a constant-time compare, and rate limiters cover read/write/vote.

## High severity

### H1. Anonymous packs are not anonymous at the API level

> **Status: fixed.** PR #254 (2026-07-13) enforces anonymity server-side —
> list/detail responses redact `author_name`/`author_id`, `?author=` excludes
> anonymous packs for non-owners, and the D1 mirror stores `Anonymous`. The
> description below records the pre-fix behaviour.

`is_anonymous` is stored on the pack
(`backend/eso-packs-worker/src/index.ts:332`) but the worker never redacts
`author_name` or `author_id` from responses: `GET /packs` returns full pack
objects (`index.ts:254`) and `GET /packs/:id` returns the pack verbatim
(`index.ts:274`). Anonymization happens only in the Rust client — and even
there only partially: `src-tauri/src/commands.rs:6338` masks `author_name` to
`"Anonymous"` but passes `author_id` through untouched.

Consequences:

- Anyone calling the public API directly (curl, a competing client) sees the
  real esologs `author_name` and `author_id` of every "anonymous" pack.
- The `?author=<id>` list filter (`index.ts:212-215`) returns a user's
  anonymous packs alongside their public ones — a second de-anonymization
  vector that works even through the official client's data model.
- The D1 mirror stores the real `author_name` for anonymous packs
  (`index.ts:51-52`), so anything reading that mirror inherits the leak.

**Recommendation:** enforce anonymity server-side. In every public response
path (list, get, share resolution, D1 mirror if it feeds anything
user-visible), when `is_anonymous` is set, replace `author_name` with
`"Anonymous"` and blank/omit `author_id` — unless the bearer token belongs to
the author (the drafts and `?author=` "my packs" flows need the real values for
the owner). Exclude anonymous packs from `?author=` results for non-owners.
Add a worker regression test asserting redaction; today no test covers it
because the behaviour doesn't exist.

### H2. Releases publish with zero verification

`.github/workflows/release.yml:42-69` runs only the `tauri-action` build and
publish on any `v*` tag — no tests, no clippy, no lint, no audit. A tag pushed
on a commit that never passed CI produces a signed, auto-updating release
shipped directly to every user. `ci.yml` itself is strong (tsc, ESLint,
Prettier, rustfmt, clippy `-D warnings`, cargo-audit, worker + frontend vitest,
`cargo test`, full build), which makes the gap purely a wiring problem.

**Recommendation:** gate the release job on the same check suite — either run
the `ci.yml` steps as a prerequisite job in `release.yml`, or require the
tagged commit to have a successful CI run (e.g. a `workflow_run`/status check)
before `tauri-action` executes. Minor related nit: the release body hardcodes
"Known issues: None known" for every release.

## Medium severity

### M1. No sanitization of user-generated Pack Hub content

`title`, `description`, `tags`, addon `name`/`note`, and `author_name` are
length-checked (`backend/eso-packs-worker/src/validate.ts`) but never
sanitized or character-restricted. `author_name` in particular is an esologs
display name — fully attacker-controlled — stored at creation
(`index.ts:331`) and served to all viewers, and mirrored into D1. XSS safety
currently depends entirely on every downstream renderer escaping correctly
(the Kalpa frontend does; future consumers of the API or D1 mirror may not).
Consider a conservative character allowlist or normalization pass at write
time for a public UGC surface.

### M2. Every authenticated worker request round-trips to esologs.com

`validateBearerToken` (`backend/eso-packs-worker/src/shares.ts:69-95`) calls
the esologs GraphQL API on every create/update/delete/vote/share request, with
no caching. This couples Pack Hub availability and latency to a third party,
and a burst of authenticated traffic amplifies onto esologs — if esologs
rate-limits Cloudflare's egress, all Pack Hub auth breaks at once. A
short-TTL (1–5 min) cache keyed on a hash of the token would remove almost all
of these round-trips without meaningfully extending revocation windows.

### M3. ESLint is not type-aware and ignores real source files

`eslint.config.js` uses only `js.recommended` + `tseslint.recommended` +
react-hooks — no `recommendedTypeChecked`, so `no-floating-promises` /
`no-misused-promises` never run, in a codebase full of fire-and-forget
promises. It also ignores three real source files beyond the vendored
`animate-ui/` directory (`eslint.config.js:24-28`: `use-auto-height.tsx`,
`use-is-in-view.tsx`, `get-strict-context.tsx`), and all of `src-tauri/` is
outside any JS lint scope (fine) but the worker has its own config only via
its package. Enabling type-checked rules will surface real bugs cheaply.

### M4. Silent error swallow on live-session stop

`src/components/uploader/use-live-session.ts:269`:
`void invokeOrThrow("uploader_stop_live", …).catch(() => {})` — a failed stop
leaves the backend live session running with no user feedback and no log.
(The nearby `.catch(() => false)` calls at `:288` and `:306` are deliberate,
commented fail-closed patterns and are fine.) At minimum log the failure; ideally
surface a toast since a stuck live session keeps tailing the log file.

### M5. E2E tests are soft and never run in CI

The Playwright specs (`e2e/`) drive a live app over CDP and are developer-only
— no CI job runs them. Worse, many assertions are conditional no-ops: the
`.isVisible().catch(() => false)` + `if (visible) expect(...)` pattern
(`e2e/addon-list.spec.ts:53-94`, `e2e/app-launch.spec.ts:52-57`, the entire
settings flow inside `if (settingsClicked)` in
`e2e/settings-dialog.spec.ts:71`) means a missing element silently passes.
They also rely on `waitForTimeout` sleeps. As written they provide little
regression protection; tightening the assertions matters more than CI wiring.

### M6. Stringly-typed errors throughout the Rust backend

Every command returns `Result<T, String>` (134 occurrences in
`src-tauri/src/commands.rs` alone); there is no error taxonomy for the
frontend to switch on, user-facing text is mixed with internal diagnostics,
and `format!("{e}")` interpolations occasionally embed filesystem paths in
messages shown to users. The frontend compensates with regex matching on error
strings (`src/lib/tauri.ts` `ERROR_HINTS`) — which works but is fragile
coupling. Migrating to a `thiserror` enum serialized with a stable `code`
field is the highest-value backend refactor available. Related:
`panic = "abort"` in the release profile (`src-tauri/Cargo.toml:86`) plus
`.expect()` on HTTP-client builders at startup means an init failure aborts
the process with no error surface.

### M7. Toolchain is ahead of its plugin ecosystem

CI and release both need `npm ci --legacy-peer-deps` because
typescript-eslint 8.x does not support TypeScript 6 (`ci.yml:36`,
`release.yml:38`). This suppresses peer-dependency conflict detection for the
whole install. Worth revisiting whenever typescript-eslint catches up, and
removing the flag the moment it does.

## Low severity / improvement opportunities

### Worker

- **Install counter is inflatable**: the one-install-per-IP-per-hour guard is a
  KV read-before-write (`index.ts:530-534`); KV is eventually consistent, so
  concurrent requests race past it. Counts are cosmetic, so this is low
  impact. (Votes are safe — idempotent toggle inside the Durable Object.)
- **Share-code enumeration**: `GET /shares/:code` is unauthenticated with
  6-char codes over a 30-char alphabet, guarded only by the 60/min/IP read
  limiter. Acceptable with the 7-day TTL, but two more characters would end
  the conversation.
- **CORS cache correctness**: the cached default `/packs` view stores an
  origin-reflected `Access-Control-Allow-Origin` with no `Vary: Origin`
  (`index.ts:256-258`, `cors.ts:23-25`), so one origin's ACAO can be served to
  another. All allowed origins are localhost/tauri, so impact is negligible,
  but it is technically wrong.

### Frontend

- **God components**: `src/App.tsx` (1,613 lines, ~40 `useState` hooks,
  prop-drilled), `src/components/uploader/uploader-workspace.tsx` (2,844),
  `src/components/saved-variables.tsx` (2,409), plus five more files over
  900 lines. Extracting App state into feature hooks (`useAddons`,
  `useUpdateFlow`, `useDeepLinks`) is the highest-leverage frontend refactor.
- **Unvirtualized big renders**: the SavedVariables tree renders recursively
  into plain scroll containers — a large `.lua` SV file is the most likely
  perf cliff. `backups.tsx` and `pack-my-packs.tsx` lists are also
  unvirtualized (addon list and discover panel already use
  `@tanstack/react-virtual` correctly).
- **Repeated listener boilerplate**: the `listen()` + `disposed` flag +
  cleanup-array pattern is duplicated across `App.tsx`, `addon-detail.tsx`,
  and `uploader-shared.tsx` — a small `useTauriEvent(name, handler)` hook
  would remove it.
- **30 react-hooks eslint-disables** (`set-state-in-effect`,
  `exhaustive-deps`), concentrated in `saved-variables.tsx`, `packs.tsx`, and
  `split-workbench.tsx`. Each `exhaustive-deps` disable is a potential
  stale-closure bug and deserves an individual justification comment.
- **No component-level tests**: the 21 test files cover extracted pure logic
  well, but `App.tsx` dialog orchestration, deep-link handling, and the batch
  update flow have zero rendering/interaction tests.

### Rust

- **Hand-rolled Win32 FFI** for ESO-process detection
  (`commands.rs:8318`): manually declared `CreateToolhelp32Snapshot` /
  `PROCESSENTRY32W` despite the `windows 0.61` crate already being a
  dependency. Correct as written, but the typed bindings would remove a
  hand-maintained `unsafe extern` block.
- **Hard-coded Windows separator**: `resolve_addon_file_path` does
  `relative_path.replace('/', "\\")` (`commands.rs:3213`) — harmless today
  (Windows-only app) but will break file-editor commands on any future
  macOS/Linux build.
- **Dormant flag**: `uploader/commands.rs:1273` `TODO(manual-stop)` — the
  one-shot manual-stop flag is never set; wire it up or remove it. (This is
  the only TODO/FIXME in the entire repository.)
- **No upload retry**: the native esologs upload path is explicitly one-shot
  (`uploader/native/client.rs:1015`). For multi-GB logs on flaky connections,
  a resumable/retrying chunk path would be a meaningful reliability win.

### CI / docs

- **Monolithic CI job**: one serial Windows job runs everything
  (`ci.yml:14`); a Prettier failure hides a Rust test failure. Splitting
  JS/Rust/worker into parallel jobs gives fail-fast signal and shorter
  wall-clock.
- **Stale agent guide**: `claude.md:9` describes Kalpa as "functional alpha"
  while README, the release notes, and all three version manifests say beta
  (`0.1.0-beta.9`).
- **`strip` is a no-op** on the only shipped target (noted in
  `Cargo.toml`'s own comment) — release binaries aren't actually stripped.

## Checked and cleared (non-issues)

Recorded so future audits don't re-litigate them:

- `reqwest 0.13.4` and `lucide-react 1.22.0` both resolve from the official
  registries (verified in `Cargo.lock` / `package-lock.json`) — unusual-looking
  version lines, but real.
- No TLS-verification bypass anywhere (`danger_accept_invalid_certs` absent);
  all HTTP clients set timeouts; addon downloads are allowlisted to
  `cdn.esoui.com` / `www.esoui.com` before fetching.
- D1 statements are fully parameterized — no SQL injection surface.
- The many `panic!`/`unwrap()` hits in `uploader/native/` are inside
  `#[cfg(test)]` validation harnesses, not reachable from user input.
- Workflow secrets are only referenced in `env:` of trusted first-party steps;
  no `pull_request_target`, no fork access to the Cloudflare token.
